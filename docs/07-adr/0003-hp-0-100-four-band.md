# ADR-0003：HP 0-100 + 4 段视觉带 + 进退阶机制

- 日期：2025-08
- 状态：accepted
- 决策人：xin（产品 + 工程）

## 背景

mealmate 的核心游戏化机制是"陪伴角色的 HP"：HP 满了角色升级（阶段进阶），HP 空了角色受伤（阶段降级）。要决定：
- HP 范围 / 粒度
- 视觉表达（数字？心？进度条？）
- 增减规则（餐 +5 / -10 是怎么定的）
- 满 / 空时的处理

## 决策

- **HP ∈ [0, 100]**
- **视觉 4 段心**：每段 25 HP，4 颗心代表四段；当前血量带渐变
- **餐打卡 +5 HP**；**错过餐 -10 HP**（gentle mode 半成）；**加餐 +10 HP**；**体重打卡 +0.5 HP**
- **HP 满 100** → advanceStage（stage+1，HP 重置 50，companionLv+1，弹完成屏）
- **HP 降到 0 以下** → demoteStage（stage-1，HP=90，弹回到屏；stage 1 时 stay 1 + support 调）
- **stage 1 init HP=60**（缓启）；**stage 2-5 init HP=50**
- 统一通过 `addHp(delta)` action 走边界，确保不变量

## 理由

- **0-100 是直觉数字**：用户不用学新概念
- **+5/-10 的比例**：错过餐扣分要大于一次拍照获益，否则用户每天只吃 1 餐也能维持
- **gentle mode 半成**：照顾敏感用户（PRD §11.L 安全条款），不是惩罚就是关怀
- **4 段视觉**：从 25/50/75/100 给用户清晰的阶段感，单数字 0-100 反而不直观
- **满 100 而非满 4 段升级**：rebase 到 50 留有缓冲，不会"刚升级就被一次错过餐打回原形"
- **0 以下重置到 90**：让用户能立刻看到"已经回血了"，避免再陷入负面感

## 数值具体表

| 动作 | HP delta | 备注 |
|---|---|---|
| 正餐拍照 | +5 | 一日 3 餐封顶 +15 |
| 加餐拍照 | +10 | 一日 2 次封顶 +20 |
| 错过餐 | -10 | gentle: -5 |
| 体重打卡（stage 2+） | +0.5 | 每日 1 次 |
| 阶段进阶后 | reset → 50 | stage 2-5 init |
| 阶段降级后 | reset → 90 | 大幅回血 |
| Stage 1 init | 60 | 缓启 |
| Stage 1 demote（HP→0） | hp=90, stay stage 1 | support 调 |

## 不变量

- `hp ∈ [0, 100]`（addHp 内 clamp）
- 单次 delta 绝对值 ≤ 10（保证不会一步从 100 跨到 -10 触发双判定）
- HP=100 必触发 advance（满 100 不停留）
- HP<0 必触发 demote（不停留，立刻 reset 90）

## 取舍

接受的：
- 4 段心比单条进度条占更多空间 → 视觉权重高，强调
- gentle mode 实现复杂度（多一层条件）
- +0.5 HP 当前用 `__dev_setHp(before + 0.5)` 临时方案

放弃的：
- 多维属性（"心情" / "活力" / "营养"）—— 太复杂，单 HP 足够表达
- 用 XP 升级而非 HP 满 → HP 双向（升 + 降）比单向 XP 表达"日复一日的努力 + 偶尔失误"更准

## 后果

- 用户测试反馈：4 段心容易理解
- Stage 1 → Stage 2 转换有真实业务接入（拍 4-5 餐左右进阶）
- Stage 2 → 3+ 的"进阶条件"PRD 还在决策（不只是 HP 满，可能要"3 天体重达标"等）
- demote stage 1 走 support 调（建议医生）是 PRD §11.L 的硬约束
