## 一、一句话概述

mealmate 是一款**陪伴式的饮食习惯养成 App**：由一只有"HP（体力值）"的机器人吉祥物陪你吃饭——你按时吃、吃饱了，它就更有精神；你跳过或吃得太少，它就变虚弱、撒娇、提醒你。通过五阶段递进目标，帮助用户从"能吃上三餐"一路走到"健康增重"和"长期坚持"。

---

## 二、产品背景 & 目标用户

### 2.1 背景

现代生活节奏里，"忘了吃"、"不想吃"、"一天只吃一顿"是真实存在的长期问题。对部分人群（长期食欲差、需要增重、康复期人群等）来说，传统计步、减脂类 App 完全不适配，他们需要的是**向内填补**而不是"管控"。mealmate 想做的是：用一个会撒娇、会担心你的小伙伴，把"按时吃饭"变成一件有反馈、有情绪、有陪伴的事情。

### 2.2 目标用户

- 长期饮食不规律、经常跳过正餐的人群；
- 处于恢复期、需要重新建立食欲和三餐节奏的人；
- 偏瘦、希望健康增重的人；
- 对"硬核数据管理"有抵触，但对"养一只小机器人陪自己吃饭"有兴趣的用户。

> ⚠️ **敏感用户提示**：这一用户群与进食障碍康复人群存在重叠。产品在机制设计与文案上必须谨慎，详见第八章「安全与伦理边界」。

---

## 三、核心价值 & 核心机制

### 3.1 核心价值

**让"好好吃饭"从任务变成陪伴**——通过一个情感化的小机器人伙伴，把进餐行为转化为有即时情绪反馈的互动，降低坚持的心理门槛。

### 3.2 核心机制：机器人 HP

- 机器人拥有一条 HP 数值；
- 用户的进餐行为（是否按时吃、是否吃饱、是否规律运动等）会**增减 HP**；
- HP 在不同区间，机器人会展现不同的情绪、文案和动画；
- 每个阶段设定一个 HP 阈值；达到阈值解锁下一阶段与成就。

HP 是**情感驱动器**而非纯数据记分板——它的目的是让用户在意"机器人今天开心吗"，而不是在意"我今天扣了几分"。

## 四、五阶段详细设计

### 阶段一：坚持（Stage 1 — 坚持吃掉三餐）

**目标**：让用户先养成"一天三顿都吃"的行为。

| 项 | 设计 |
|---|---|
| 初始 HP | 10 |
| 跳过一餐 | HP -1 |
| 吃一餐 | HP +0.5 |
| 阶段升级阈值 | HP 达到 22 → 解锁阶段二 |
| 阶段成就 | 「可以坚持吃掉三餐」 |

**HP 区间与机器人行为：**

- **HP < 5**：机器人变得黏人，用情感化方式提醒用户"再不吃我就要不见了哦"（loss-framed 威胁的表达要谨慎，见第八章）。
- **HP 5–10**：机器人轻声说自己饿了，想再吃点东西。
- **HP 10–13**：机器人说自己在回复体力了，想继续下去。
- **HP 13–16**：机器人开心，鼓励用户继续冲到 22，解锁下一阶段。
- **HP 16–22**：（补齐区间）继续保持、即将升级。

**提醒 & 验证流程：**

1. 按固定时间（用户可在设置里自定义三餐时段）推送进餐提醒；
2. 推送后进入**验证窗口**（默认 30–60 分钟，时长待定）；
3. 窗口内用户需**上传一张餐食照片**作为吃饭凭证；
4. 窗口结束：
   - 未上传 → HP -1，并发送一条温和的"下一顿我们一起加油"消息；
   - 已上传 → HP +0.5，机器人表达肯定与鼓励。
5. HP 累积到 22，进入阶段二。

---

### 阶段二：量化（Stage 2 — 吃饱三餐）

**目标**：从"吃上"进化到"吃饱"，并引入体重追踪。

| 项 | 设计 |
|---|---|
| 初始 HP | 10 |
| 跳过一餐 | HP -1 |
| 吃一餐（含饱腹度记录） | HP +0.5，且饱腹度 ≥ 7 才有效 |
| 阶段升级阈值 | HP 达到 31 → 解锁阶段三 |
| 阶段成就 | 「可以吃饱三餐」 |

**硬性时间窗**：

- 早餐必须在 **09:00 前**完成；
- 午餐必须在 **14:00 前**完成；
- 晚餐必须在 **18:00 前**完成；
- 超出时间窗视为跳过该餐。

**饱腹度机制**：

- 每餐除了上传照片外，需要填写**饱腹度评分 0–10**；
- 其中 10 = "此生吃过最饱的状态"；
- 评分 ≥ 7 → 该餐记为"吃饱"，HP +0.5；
- 评分 < 7 → 不加 HP（但也不扣，只是不计入进度）。

**每日称重**：

- 每天 **21:00 前**需上传一张体重秤照片；
- 未上传 → HP -1；
- 已上传 → HP +0.5。

**每周报表**：

- 每周自动生成报告，包含体重变化折线图、吃饱率、HP 曲线等；
- 由机器人以"我的小结"口吻呈现。

HP 累积到 31 → 解锁阶段三。

---

### 阶段三：健康增重（Stage 3 — 吃饭、运动两不误）

**目标**：在吃饱基础上，引入规律运动，目标是**健康增重**（而非单纯体重上升）。

| 项 | 设计 |
|---|---|
| 初始 HP | 43 |
| 跳过一餐 或 饱腹度 < 7 | HP -1 |
| 吃一餐 | HP 不变（阶段三里"吃"是基本盘，不再加分） |
| 每周体重净增 1kg | HP +4 |
| 运动：每周 3 次、共 3 小时 | 每次完成 +1，当周未运动 -1 |
| 阶段升级阈值 | HP 达到 50 → 解锁阶段四 |
| 阶段成就 | 「吃饭、运动两不误」 |

**机制解读**：

- 之所以初始 HP 从 43 起步（而非 10），是因为此阶段失分项更多、加分项更稀（每餐不再加分，主要靠每周增重和每周运动加分），需要给用户一个缓冲垫。
- "每周体重 +1kg → +4 HP" 是最大加分项，同时也是**潜在风险点**（见第八章，避免变相鼓励过快增重）。
- 运动是新引入的行为维度，每周 3 次、共 3 小时的结构既不激进也足以形成习惯。

---

### 阶段四：营养（Stage 4 — 待定）

**状态**：待定 / To Be Defined

用户尚未给出阶段四的具体机制。以下为**建议候选方向**（仅作为讨论起点，尚未决定）：

- **候选 A — 三大宏量追踪**：蛋白质 / 碳水 / 脂肪按日达标比例转化为 HP。
- **候选 B — 食物种类多样性**：每日不同食物类别（谷物、肉蛋、蔬、果、奶、豆、坚果）覆盖度。
- **候选 C — 轻量级标签法**：用户为每餐打"主食 / 蛋白 / 蔬果"等标签，不强制量化。

待决定事项：是否引入食物识别 AI？是否允许用户手动填写营养标签？是否接入第三方食物数据库？

## 五、关键交互

### 5.1 推送提醒

- 三餐定时推送（本地通知为主，必要时辅以云端推送）；
- 推送文案由机器人情绪驱动——HP 低的时候语气更弱/更黏人，HP 高的时候更活泼；
- 用户可在设置里调整三餐时段、关闭某一餐的提醒、切换文案风格（见第八章）。

### 5.2 拍照验证

- 点击推送直接进入拍照页；
- 支持现场拍摄和相册选取；
- 上传后立即看到机器人情绪反馈（+0.5 HP 对应的动画 / 台词）。

### 5.3 饱腹度评分（阶段二起）

- 餐食照片上传后的下一步，0–10 滑块 + 表情提示；
- 附带轻量说明"10 = 你有史以来最饱的一次"。

### 5.4 每日称重（阶段二起）

- 21:00 前完成；
- 拍摄体重秤照片 + 手填当前体重（后续可考虑 OCR 自动识别秤面数字）。

### 5.5 每周报表

- 每周固定日自动弹出；
- 包含：HP 曲线、每餐按时率、吃饱率、体重折线、运动达成率（阶段三起）；
- 语气为机器人的"小结/日记"。

### 5.6 机器人主屏

- App 首屏就是机器人；
- 根据 HP 区间显示不同动画 / 表情 / 文案；
- 点击机器人可查看详细数据。

## 六、MVP 范围建议

**优先打磨阶段一的完整闭环**，让它能在 TestFlight 上跑完整张"坚持吃三餐"的旅程。

MVP 必须包含：

1. 账号 / 基本设置（三餐时段、时区）；
2. 机器人主屏 + HP 显示 + 5 个区间的表情/文案；
3. 三餐本地通知推送（iOS + Web Push）；
4. 拍照验证流程（相机 + 相册 + 上传）；
5. 验证窗口打点与加减 HP 逻辑；
6. 阶段一到阶段二的解锁流程 + 成就展示；
7. 最小数据后端（用户、餐食记录、HP 事件流）。

MVP 暂不包含：

- 饱腹度评分、每日称重、每周报表（阶段二起引入）；
- 运动记录（阶段三起引入）；
- 营养追踪（阶段四起，且尚未定义）；
- 好友 / 社交、对外分享；
- Apple Watch / Widget / Live Activity；
- 付费订阅。

> 阶段二~五在 MVP 验证后按顺序迭代；每阶段独立发版，避免一次性堆过多机制。

## 七、技术选型（摘要）

详细比较与来源见 [`./tech-research.md`](./tech-research.md)。

**推荐：React Native + Expo + React Native Web**

理由概要：

- 单一 TypeScript 代码库覆盖 iOS 与 Web；
- `expo-notifications`、`expo-image-picker`、`expo-camera` 直接命中本产品最核心能力（定时提醒 + 拍照验证）；
- EAS Build 一键上 App Store；EAS Update 支持 OTA 热更，适合频繁微调 HP/文案/机器人情绪参数；
- 招聘 / 外包成本最低；长期维护稳健。

**次选方案**：Flutter + Flutter Web——当对机器人动画像素级一致性要求极高、或团队已深耕 Dart 时可考虑；但其 Web 端 SEO 与首屏相对弱。

**明确不推荐**（此项目场景下）：

- **纯 PWA**：iOS 推送与后台限制无法承载"定时情感化提醒"。
- **Capacitor / Ionic**：对情感化互动 App，WebView 层动画手感不足，且纯套壳更容易遇到 App Store 审核问题。
- **SwiftUI + Next.js 两套码**：体验上限高，但小团队早期工时代价过大；可作为未来 v2 迁移方向。

## 八、安全与伦理边界

mealmate 的目标用户中，会有一部分是进食障碍康复期、重度焦虑、或对身体数据敏感的人群。对这些用户，「HP 归零 / 机器人消失 / 你让它失望了」这类 loss-framed 威胁，可能从"情感激励"滑向"负向反馈循环"，加剧自责与失控感。相关风险必须在产品设计中显式处理。

**已识别的风险点：**

- 阶段一中"机器人会消失"的威胁叙事；
- 阶段二"硬性时间窗未达 → 扣 HP"容易转化为"我又没做到"的自罪感；
- 阶段三"每周 +1kg → +4 HP"的强正反馈可能诱导过快增重，不符合"健康"初衷；
- 把体重数字、饱腹度数字做成唯一衡量尺，对数字敏感用户不利。

**设计规范（必须执行）：**

1. **永远不把 HP 下降描述为"你失败了"**——文案统一表达为"机器人今天没什么精神，我们明天再一起加油"这类陪伴型语气；
2. **提供"温柔模式"开关**（设置里的 tone 切换）：开启后，所有 loss-framed 文案（包括"我会消失"）替换为温和版本；默认对未成年账号或首次使用账号启用温柔模式；
3. **体重目标设上限**：阶段三的 +1kg/周 设为上限而非目标，超过 1.5kg/周时机器人主动建议放慢；
4. **不显示绝对体重排行榜 / 对比**，不引入社交减肥式比较；
5. **设置里提供"暂停一切提醒"与"导出/删除我的数据"**两个一等入口；
6. **正式上线前，在欢迎页与隐私政策中加入简短提示**，建议有进食障碍、焦虑或情绪困扰的用户在咨询专业人士后再决定是否使用，并在应用内常驻入口指向相关公共资源。

这些边界不是锦上添花，而是该产品能否对真实用户负责的底线。

## 九、非目标 / 待决策项

**本版本明确非目标：**

- 不做减脂 / 卡路里管控，避免与目标人群反向；
- 不做社交排行榜；
- MVP 不做付费订阅。

**待决策（需要进一步讨论）：**

1. **阶段四（营养）** 机制待定；
2. **阶段五（持之以恒）** 机制待定；
3. **文案 tone 基线**：默认语气偏软还是偏俏皮？需要 2–3 个候选 voice 让真实用户试读；
4. **验证窗口时长**：30 分钟 vs 60 分钟 vs 用户自定义？
5. **是否允许补录**：用户过了窗口但确实吃了饭，能否补传照片换回部分 HP？
6. **体重识别**：手填 vs 秤面 OCR？MVP 建议手填 + 照片凭证即可；
7. **食物识别 AI**：阶段四是否必需？若需要，是本地模型还是云端？
8. **后端**：Supabase / Firebase / 自建；
9. **隐私与合规**：照片与体重是高敏感数据，需明确存储位置、留存周期、删除机制；若面向中国大陆需走 ICP/个保法，海外需 GDPR/CCPA；
10. **账号体系**：手机号 / Apple ID / 邮箱的优先级；
11. **Apple Watch / Widget / HealthKit**：列入 v2 还是先不做？这会影响技术栈上是否给 Swift 模块留口子。

### 11.B Stage 2 主页设计（按 Figma 1:171 = 8303:15）

视觉重做。组成（从上到下）：

1. **状态大标题**（按 HP band 动态切）：
   - HP 100：「满血状态」
   - HP 60–99：「状态平稳」
   - HP 30–59：「状态欠佳」
   - HP 10–29：「非常虚弱」
   - HP 0–9：「濒临耗尽」  
   每条配 2 行副标（mascot 口吻，待文案细化）
2. **HP 心形条**：10 颗心 icon，按当前 HP/100 等比填充（1 心 = 10 HP）+ 右侧 `X/100` 数字
3. **当前体重模块**（xin 明确加在首页）：
   - 大字最近一次体重（`60.5 kg`）
   - 副标 `对比上次 ±X.X kg · MM-DD HH:mm`
   - 点击进 `weight-entry` modal
4. **下一餐倒计时卡片**：
   - 标题动态切：「早餐时间到啦!」/「午餐时间到啦!」/「晚餐时间到啦!」/「明天早餐」
   - `mm:ss` 大字（距离时间窗末，按 settings 里设的提醒时间 ±90min 算）
   - 主 CTA：「去拍照」→ photo modal

### 11.F 餐后消息生成规则（v0.4 核心业务规则，新增）

每一餐有时间窗 = settings 里设的提醒时间 ±90 分钟。两条结局：

#### 11.F.1 拍照通过（窗内拍照 + 走完 photo 闭环）

- 自动 push 进 `dialogueHistory` 两条：
  1. 「太棒了！{早餐|午餐|晚餐}看起来不错」（按当前 mealSlot 切；从 mock 池抽具体话术变体）
  2. 一条鼓励话（「继续加油哈，这么下去一定可以尽快达到目标的!」同款，从 mock 池抽）
- 两条都 visible 在记录页 feed

#### 11.F.2 错过时间窗（窗末未拍照）

- HP **-15**（v0.1 是 -1，v0.4 调到 -15；详 §11.I）
- 首页自动切到"状态不好"变体页（按 HP band 自然切，已经有 band 逻辑）
- 自动 push 两条：
  1. 「你错过了一餐，有点饿」（带 -15 标签）
  2. 「第二阶段贵在坚持，希望下一顿可以吃上饭」（stage 1/2 文案不同；stage 1 改成「坚持每天三餐才是开始呀」）

#### 11.F.3 触发机制

- 每餐时间窗末用 `expo-notifications` schedule 一个 silent local 通知（dataOnly，不弹横幅）
- silent 通知触发时（app 在后台也能收）跑 missed-check：检查该餐是否有 mealRecord 且时间在窗内 → 没有就走 §11.F.2
- 真机和 simulator 都能跑（B1 已经接好的工具链）
