# v1.2.1 — Stage 0 / 0.5 缓冲层

> Status: **设计中**(2026-06-01 草案,等 xin 拍 OPEN-1/2/3/4 后实施)
> Branch: `feat/v1.2.1-stage-0`
> Target: v1.2.1 上 TestFlight

---

## 1. 目标

降低 Stage 1 的进入门槛,避免「不吃早餐 → 第一次错过 → -10 HP → 觉得失败 → 卸载」的劝退路径。

观察:v1.0/1.1 现有逻辑直接 onboarding 完落到 Stage 1(HP 60),错过早餐 -10 → HP 50,但用户毫无心理准备。

**新流程**:`onboarding → Stage 0(零门槛) → Stage 0.5(纯鼓励) → Stage 1(正常机制)`

## 3. 阶段规格

### Stage 0 — 试一下(零门槛)

| 属性 | 值 |
|---|---|
| 进入 | onboarding 完(`onboardingDone=true`)或 v1.2.1 migrate 触发(老用户)|
| HP 概念 | **不显示**(home 不画 HP 条;HP store 值保持 100 placeholder)|
| 漏餐 | **无影响**(missedScan 不触发 → HP 不扣 / dialogue 不 push「错过」)|
| 唯一通关条件 | 拍 1 张餐照,Food-101 backend 验证 `is_food=true` → 自动进 Stage 0.5 |
| dialogueHistory | 只 push 1 条 stage_0_done 庆祝 + transition pending |
| 加餐入口 | **不显示**(避免分流)|
| 体重秤入口 | **不显示**(stage 0 还不到聊体重的阶段)|
| WeekStrip | **不显示** |
| 餐时段卡 | **不显示** |

### Stage 0.5 — 起步鼓励(0/40 分)

| 属性 | 值 |
|---|---|
| 进入 | Stage 0 拍完那张照 → `currentStage = 0.5, stage05Score = 0` |
| 分数 | 0 → 40 通关 |
| **+10** | 餐照打卡 OR 加餐照(都通过 Food-101 严格验证) |
| **+0** | 漏餐 / 网络失败 / 食物验证拒绝 |
| **-N** | **从不**减分(纯鼓励阶段,无惩罚)|
| 通关 | `stage05Score >= 40` → 自动进 Stage 1 + 满 100 HP |
| 时长上限 | **无**(就算 30 天没吃 40 顿也不会 demote / reset)|
| 漏餐显示 | WeekStrip cell 显示为**灰色**(不变红),配 mascot「没吃也没关系」文案 |
| 加餐 | **算分**(每顿都 +10,**不**复用 Stage 1 的 SNACK_DAILY_LIMIT=3)。设新常量 `STAGE_05_SNACK_DAILY_LIMIT`,建议 **= 3**(避免一天点 4 次加餐 hack 通关 → 实际配合正餐 3 + 加餐 3 = 6 次/天 = 60 分 = 一天通关上限不变,但更宽容,但是不暴力)。**OPEN-4** 待 xin 拍 |
| 体重秤 | **可显示**(让用户养成习惯,但不影响分数)|
| WeekStrip | **显示**(让用户感受日历感),用灰色 cell 不强调 |
| 餐时段卡 | **显示**,漏餐不变红 |

### Stage 1+ — 不动

完全沿用现状(HP 0-100 / 餐 +5 / 漏 -10 / advance / demote / stage 2-5 全部机制)。Stage 0.5 → 1 通关时:
- `currentStage = 1`
- `hp = 100`(满血开局,跟 v1.0 onboarding 完一样)
- push transition pending `{stage:1, kind:'start'}`
- `stageHistory` append

## 4. Store schema 改动(v12 → v13)

### 4.1 字段变化

| 字段 | v12 (旧) | v13 (新) |
|---|---|---|
| `currentStage` | `1 \| 2 \| 3 \| 4 \| 5` | `0 \| 0.5 \| 1 \| 2 \| 3 \| 4 \| 5` |
| `stage0Done` | — | `boolean`(Stage 0 拍照后置 true,持久化但只 stage 0 用)|
| `stage05Score` | — | `number`(0-40)|
| `archive_v12` | — | `Array<{ archivedAt: number, snapshot: <whole v12 state> }>`(用数组,允许多次归档但通常只有一次)|

### 4.2 migrate 算法 v12 → v13

```ts
// app/src/store/useStore.ts migrate function
if (version === 12) {
  const archivedAt = Date.now();
  const snapshot = { ...state }; // 完整 dump

  // 保留的用户属性(不属于"进度"的字段)
  const preserve = {
    onboardingDone: state.onboardingDone,
    onboardingCompletedAt: state.onboardingCompletedAt,
    mealSchedules: state.mealSchedules,
    robotName: state.robotName,
    gentleMode: state.gentleMode,
    height: state.height,
    gender: state.gender,
    ethnicity: state.ethnicity,
    targetWeight: state.targetWeight,
    skipWeightPhoto: state.skipWeightPhoto,
  };

  // reset 到 stage 0(保留用户属性,清进度)
  return {
    ...preserve,
    // v13 新字段
    currentStage: 0,
    stage0Done: false,
    stage05Score: 0,
    archive_v12: [
      ...(state.archive_v12 ?? []),
      { archivedAt, snapshot },
    ],
    // 进度全清
    hp: 100,
    todayMeals: { breakfast: "pending", lunch: "pending", dinner: "pending" },
    todayKey: getTodayKey(),
    mealHistory: {},
    mealRecords: [],
    weightHistory: [],
    fullnessHistory: [],
    hpHistory: [],
    dialogueHistory: [],
    stageHistory: [{ ts: archivedAt, stage: 0, reason: "v13_migrate" }],
    stage5StartedAt: null,
    stage5Stars: 0,
    stage5LastStarCheck: null,
    transitionsSeen: [],
    transitionsPending: [{ stage: 0, kind: "start" }], // 弹一次「这里慢慢来」开场
    disappearWarningLastShownAt: null,
  };
}
```

### 4.3 onboarding 完成的初始 state(新用户)

新装的 v1.2.1 用户:
- `currentStage = 0`
- `stage0Done = false`
- `stage05Score = 0`
- `archive_v12 = []`(空,他们没历史)
- `transitionsPending = [{stage:0, kind:'start'}]`

## 5. 后端归档机制

### 5.1 新表 `user_archives`

```sql
CREATE TABLE IF NOT EXISTS user_archives (
  id              INTEGER PRIMARY KEY AUTOINCREMENT,
  user_id         TEXT NOT NULL REFERENCES users(id) ON DELETE CASCADE,
  schema_version  INTEGER NOT NULL,
  payload         TEXT NOT NULL,
  archived_at     TEXT NOT NULL
);

CREATE INDEX IF NOT EXISTS idx_user_archives_user_id ON user_archives(user_id);
```

### 5.2 `POST /sync/push` 改动

接受新字段 `client_schema_version`(int,客户端传当前 schema 版本)。逻辑:

```py
@app.post("/sync/push")
def sync_push(body: SyncPushBody, user_id: str = Depends(require_user)) -> dict:
    # 1. 如果 body.schema_version <= 12,自动归档当前 user_data.payload 到 user_archives
    #    (这是用户 client v1.2.1 启动后第一次 sync_push,自动留底)
    if body.schema_version <= 12:
        with get_conn() as conn:
            old = conn.execute(
                "SELECT payload, schema_version FROM user_data WHERE user_id = ?",
                (user_id,),
            ).fetchone()
            if old:
                conn.execute(
                    "INSERT INTO user_archives (user_id, schema_version, payload, archived_at) VALUES (?, ?, ?, ?)",
                    (user_id, old["schema_version"], old["payload"], now_iso()),
                )
    # 2. 走原 upsert 逻辑
    ...
```

注意:**backend 不强制 reset 用户进度** — 前端 zustand migrate 自动做 reset。backend 仅备份。

## 9. 待 xin 拍板的 OPEN

### OPEN-1: 「需要帮助?」link 指向哪里?

Stage 0 拍照三次拒绝后显示的 link。3 个选项:

### OPEN-2: v1.1 老用户被 reset 后,需不需要「欢迎回来」welcome modal?

migrate 自动归档 + reset 后,老用户打开 app 会看到:
- mascot 名字保留(他们起的名)
- 体重 / 身高 / 性别保留
- 但所有进度被清(从 stage 0 开始)

如果**不**加 welcome modal,他们可能误以为 app 出 bug 了 / 数据丢了。

**推荐**:**加一个 welcome modal**(在 stage_0_start transition 触发前先弹一次,仅 v1.1 → v1.2.1 migrate 用户看见 — 通过 `archive_v12.length > 0` 判定):
```

「v1.2 我们改了节奏。你之前的数据我都好好收着,从今天开始我们重新一起慢慢来 ✨」
[继续]
```

### OPEN-3: Stage 0 那张餐照算不算今天的早/午/晚?

3 个选项:
- **A. 算**(同时 `markMealDone(currentSlot)` — 但当前没有 slot 概念,要按时间窗口判定 → 7-10 算早餐,11-15 午餐,17-22 晚餐,其他 = 加餐处理)
- **B. 不算,只是 stage 0 进度标志位**(simpler;但用户可能"我拍了为啥首页还没显示我吃了早餐?")
- **C. 算加餐**(简单 — 一律 +10 加餐计数,不影响正餐 slot)

**推荐**:**B 简单先做,xin 后续可改成 A**。Stage 0 极简理念是「不引入 slot 概念」— 那张餐照就是入门礼,不进 mealRecords。

### OPEN-4: Stage 0.5 的 +10/餐,加餐有没有上限?

3 个选项:
- **A. 无上限**(每顿都 +10,理论上可以一天点 4 顿外卖通关)
- **B. 每日上限 3 加餐**(同 Stage 1 SNACK_DAILY_LIMIT=3,日上限 +30 = 跟 v1.1 一样)
- **C. 全程上限 4 次加餐**(把 stage 0.5 限制成「正餐为主」,加餐只是 4 张鼓励)

**推荐**:**B**(沿用现 SNACK_DAILY_LIMIT=3 常量),不引入新常量,Stage 0.5 跟 Stage 1+ 加餐规则一致。3 餐 + 3 加餐 / 天 = +60/天上限,3 天最快通关。

## 12. 风险 / 已知 trade-off

- **现有 stage 5 老用户 reset 觉得被惩罚** — 通过 welcome modal + 强调"数据收着"缓解;但确实是 breaking change
- **新用户 stage 0 拍照失败 3 次"需要帮助"link 是占位** — Phase 1 ship 时可能还没真 FAQ;OPEN-1 等 xin 拍
- **archive_v12 字段会膨胀 AsyncStorage size** — 整包 dump 可能 几十 KB(取决于历史长度)。AsyncStorage 6MB 上限,正常用户没事,但极端 case(2 年历史)要注意。**实施时确认下当前持久化大小**
- **stage 0 / 0.5 没 HP** — 现 `_layout.tsx:hp.ts` 各处假设 HP 1-100,要确认 stage 0/0.5 不读 HP 不会爆(写 unit test)
- **Stage 0/0.5 不显示 WeekStrip / 餐时段是 stage 0 only,stage 0.5 重新出现** — UI 切换比较突兀,过渡屏要把这个改动告诉用户
