## 1. 目标

降低 Stage 1 的进入门槛,避免「不吃早餐 → 第一次错过 → -10 HP → 觉得失败 → 卸载」的劝退路径。

**新流程**:`onboarding → Stage 0(零门槛) → Stage 0.5(纯鼓励 / 4 颗爱心) → Stage 1(正常机制)`

## 3. 阶段规格

### Stage 0 — 试一下(零门槛)

| 属性 | 值 |
|---|---|
| 进入 | **仅新用户** onboarding 完(`onboardingDone=true && currentStage===0`)|
| HP | **不显示**(home 不画 HP 条;HP store 值保持 100 placeholder)|
| 漏餐 | **无影响**(missedScan stage <1 短路 → HP 不扣 / dialogue 不 push「错过」)|
| 通关条件 | 拍 1 张餐照,Food-101 backend 验证 `is_food=true` → 自动进 Stage 0.5 |
| 那张餐照怎么处理 | **算今天最近 slot 的餐**:`markMealDone(closestSlotByWallClock())` + push 正常 meal_done dialogue;同时 `setStage0Done(true)` + `advanceToStage05()` |
| dialogueHistory | 跟现 stage 1 一致(meal_done + encourage)+ stage 0 end transition |
| 加餐入口 | **不显示**(避免分流;只让用户走那条主线)|
| 体重秤入口 | **不显示** |
| WeekStrip | **不显示** |
| 餐时段卡 | **不显示** |

### Stage 0.5 — 起步鼓励(4 颗爱心 ❤️❤️❤️❤️)

| 属性 | 值 |
|---|---|
| 进入 | Stage 0 拍完那张照 → `currentStage = 0.5, stage05Score = 0`(那张照已经在 Stage 0 计为正餐,**不**在 0.5 计爱心)|
| 爱心数 | 0 → 4(每颗 = 拍 1 张通过 Food-101 验证的餐照 / 加餐照) |
| **+1 爱心** | `markMealDone(slot)` 触发 OR `addSnack(...)` 触发(只要食物验证通过)|
| **-N 爱心** | **从不**减(漏餐 / 验证失败 / 网络挂 → 0 影响,爱心永不掉)|
| 通关 | `stage05Score >= 40`(即 4 颗满)→ stage_0_5_end transition → Stage 1 + HP 满 100 |
| 时长上限 | **无**(就算 30 天没拍 4 张也不会 demote / reset)|
| 漏餐显示 | WeekStrip cell **灰色**(不变红),mascot 鼓励文案「没吃也没关系」|
| 加餐 | **算 +1 爱心**,日上限沿用 `SNACK_DAILY_LIMIT=2`(回退后)。理论一天可拍 3 正 + 2 加餐 = 5 次,1 天内通关(只需 4 次)|
| 体重秤 | 显示(养成习惯),**不**影响爱心 |
| WeekStrip | 显示 |
| 餐时段卡 | 显示,漏餐**不变红**(灰色)|

### Stage 1+ — 不动

完全沿用现状(HP 0-100 / 餐 +5 / 漏 -10 / advance / demote / stage 2-5 全机制)。Stage 0.5 → 1 通关时:
- `currentStage = 1`
- `hp = 100`(满血开局)
- push transition pending `{stage:1, kind:'start'}`
- `stageHistory` append `{ts, stage:1, reason:'stage05_complete'}`

## 4. Store schema 改动(v12 → v13)

### 4.1 字段变化

| 字段 | v12 (旧) | v13 (新) |
|---|---|---|
| `currentStage` | `1 \| 2 \| 3 \| 4 \| 5` | `0 \| 0.5 \| 1 \| 2 \| 3 \| 4 \| 5` |
| `stage0Done` | — | `boolean`(Stage 0 拍照后置 true)|
| `stage05Score` | — | `number`(0-40,步长 10)|

**没有 archive_v12,没有 reset,migrate 只加字段。**

### 4.2 migrate v12 → v13

```ts
// app/src/store/useStore.ts migrate function
if (version === 12) {
  return {
    ...state,
    stage0Done: state.stage0Done ?? false,
    stage05Score: state.stage05Score ?? 0,
    // currentStage 保留(老用户已在 1-5,新装才会从 initialState 拿 0)
  };
}
```

### 4.3 initialState (新装 v1.2.1 用户)

```ts
const initialState = {
  ...其他原字段,
  currentStage: 0,         // ← 改自 1
  stage0Done: false,
  stage05Score: 0,
  stageHistory: [{ ts: Date.now(), stage: 0, reason: "init" }],
  transitionsPending: [{ stage: 0, kind: "start" }],   // onboarding 完弹一次开场
};
```

### 4.4 actions 新增

```ts
setStage0Done: () => void;                   // 只在 stage 0 拍完那张照调
incrementStage05Score: (delta: number) => void;  // +10/次,内含 >=40 advance trigger
advanceFromStage0: () => void;                // stage 0 → 0.5 + push end+start transitions
advanceFromStage05: () => void;               // stage 0.5 → 1 + hp=100 + push end+start
```

### 4.5 现有 actions 改动

- `markMealDone(slot, ...)`: 末尾,如果 `currentStage === 0.5` → `incrementStage05Score(10)`
- `addSnack(...)`: 末尾,如果 `currentStage === 0.5` → `incrementStage05Score(10)`
- `missedScan` / 漏餐处理:如果 `currentStage < 1` → 不扣 HP / 不 push missed dialogue(stage 0/0.5 漏餐免疫)

## 12. 风险 / 已知 trade-off

- **stage 0 / 0.5 没 HP** — `_layout.tsx` / `hp.ts` 各处可能假设 HP 1-100,要确认 stage 0/0.5 时不读 HP 不爆。strategy: missedScan 早 return,getHpBand 不被调
- **Stage 0 → 0.5 切 UI**(0 极简 / 0.5 完整)突兀 — 通过 transition 屏「下一步,集齐 4 颗爱心」缓冲
- **TS 字面量 0.5 在 union 里语义清晰但容易和 string"0.5"混淆** — 全程统一用 number,持久化也是 number
- **不动老用户的副作用**:已在 stage 5 用户不会看到 Stage 0/0.5 UI 改动(无影响,符合设计)
- **`incrementStage05Score` 跑在 markMealDone / addSnack 末尾 — 顺序敏感** — 必须在 `addHp` / `pushDialogue` 之后,因为顺序影响 stageHistory append 逻辑
